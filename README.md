## v10.5.149 Debug 阻塞分级与说明完善版

本版基于 v10.5.148 继续完善 Debug 灵敏度系统。重点不是继续修已确认可忽略的 iOS Safari 小偏差，而是把“阻塞 / 非阻塞 / 已降级 / 已忽略”的含义写清楚，并让自检输出更容易判断是否真的影响发版。

### 本版处理

1. 完善 Debug 对“阻塞”的定义：阻塞只指会影响核心功能、数据安全、版本可靠性或发布可信度的问题。
2. 严格模式仍原样显示 warning，但已知 `jump-chip` 暗夜浅色背景审计偏差、iOS Safari 顶部 16px 左右微偏移会标记为“非阻塞”。
3. 一键自检新增 `issueGroups` 分组：`blockers`、`warnings`、`tips`、`downgraded`、`ignored`、`raw`。
4. Debug 摘要文案从“阻塞”改为“阻断发布项”，同时显示非阻塞警告、已降级、已忽略数量，避免误导用户。
5. 标准模式继续把已知误报降级为 `info`；宽松模式继续弱化非核心提示；发布模式只保留真正阻断发布的问题。
6. 同步更新首页版本、版本记录、README、manifest、service-worker 与知识库版本说明，继续保持 1 个主 style + 1 个主 script。

### Debug 灵敏度说明

- 严格模式：所有 warning 原样显示，适合开发、清补丁和排查真实样式问题；已知误报会标记为非阻塞，不计入阻断发布项。
- 标准模式：推荐日常使用；保留核心错误与需要复核的风险，把已知 iOS Safari 小偏差降级为提示。
- 宽松模式：适合普通使用测试；非阻塞视觉、未初始化、本地设置类 warning 多数降级为提示。
- 发布模式：只保留会阻断发布的核心问题，其余提示放到 `ignoredIssues` / `issueGroups.ignored` 里备查。

### 什么是阻塞

阻塞 = 会影响核心功能、数据安全、版本可靠性或发布可信度的问题，例如：

- JS 运行错误
- 重复 ID
- 版本标题/主源码版本不一致
- 旧版本锁或延迟补丁残留
- 模板串用
- 导航断链
- 历史记录、分析、命盘等主命名空间缺失

普通视觉提示、已知 iOS Safari 审计偏差、本地设置未初始化，一般属于非阻塞。

### 测试建议

1. 上传全部文件后，Safari 强制刷新。
2. 打开设置 → 界面显示 → Debug 灵敏度，建议先用“标准模式”。
3. 打开 Debug → 清理旧缓存 → 再刷新一次。
4. 打开 Debug → 一键自检。
5. 如果“阻断发布项”为 0，即使严格模式仍显示少量已知非阻塞 warning，也可以视为不影响发布。
